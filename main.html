Section 02

Investigation

Researching this topic began with understanding how real-world wildfire early warning systems work. The Bosch Forest Fire Detection System [1] uses a network of environmental sensors that monitor temperature, humidity, and gas concentration. Unlike simple threshold-based alarms, these systems use machine learning models trained on historical data to detect anomalous conditions before a fire spreads. This was an important insight: reacting to a fixed temperature is not the same as detecting danger relative to context.

The WWF Guianas early warning network [2] takes a similar approach, emphasising how detecting rate of change matters more than absolute values. A forest that is normally warm will have a different danger profile to one that is usually cool — the same temperature could be perfectly safe in one and critical in the other. This directly influenced the decision to use a rolling baseline rather than a hardcoded threshold in this project.

Research into existing micro:bit environmental projects [3] confirmed that the built-in temperature and light sensors, while not research-grade instruments, are sufficient for simulating environmental monitoring logic. Several examples online used fixed thresholds (e.g. "if temperature > 30, alert") but none implemented adaptive baselines on the device itself.

The EU Climate Change website [4] provided context on how deforestation and rising temperatures interact with wildfire risk — specifically that canopy loss increases the amount of direct sunlight reaching the forest floor, drying ground vegetation and dramatically raising fire probability. This directly motivated the second what-if scenario in the Python model.

Based on this research, the key design decisions were: to use a rolling average to define "normal" dynamically, to combine both temperature and light as risk factors (since real wildfire risk depends on both heat and dry conditions), and to build the Python model as a virtual extension of the same adaptive logic running on the device.

Section 03

Plan and Design

Design Objectives

The system needed to meet all basic and advanced requirements of the brief while remaining achievable on a BBC micro:bit — a device with no network connectivity, limited memory, and a 5×5 LED display. The core design objective was to build something genuinely adaptive rather than simply reactive: a system that adjusts its own model of the environment rather than comparing readings to fixed numbers.

Options Considered

Initially, a simple threshold system was implemented (alert if temperature exceeds 35°C). This was rejected because it does not account for ambient conditions — 33°C might be normal on a hot summer afternoon but dangerous if the baseline is 20°C. The exponential moving average approach was chosen because it is computationally cheap enough to run on the micro:bit in a continuous loop, requires no extra memory storage, and produces a genuinely adaptive baseline.

A Raspberry Pi was considered as an alternative platform but rejected — the micro:bit is sufficient for the sensors required and is more relevant to the embedded systems brief.

Stakeholders and End User Needs

The primary end user is a forest ranger or environmental monitoring officer who needs to be alerted to dangerous conditions quickly and clearly. This shaped two design decisions: the LED display must be readable at a glance (simple iconic symbols rather than numbers), and the alarm must be audible. A secondary user is a data analyst who would use the logged CSV data and Python model outputs to review historical risk patterns and test scenarios. Their need is for comprehensive, well-labelled data — which is why seven variables are logged per reading including the internal model state.

Technologies

The embedded system uses MakeCode TypeScript on the BBC micro:bit, which provides built-in access to the temperature sensor, light sensor, LED matrix, buzzer, buttons, and datalogger. The computer model is written in Python 3 using only the standard library (csv, os, random, math) so it can be run without installing any packages. This ensures the examiner can run it without any setup.

Adaptive Feedback Loop Design

The core adaptive loop was designed as: Sense → Update model → Recalculate thresholds → Score risk → Adapt margins → Respond → Log → repeat. The key innovation is that the warning margins themselves change over time — sustained high risk tightens them (more sensitive), sustained calm relaxes them — meaning the system's behaviour emerges from the data it collects rather than from hardcoded rules.

Section 04

Create

Development Milestones

Week 1–2

Basic Sensor Reading and Display

Initial micro:bit code read temperature and displayed it on button press. Light level reading added. Simple threshold alert (temp > 35°C = exclamation mark) implemented and tested.

Week 3–4

Logic Bug Fixes

Three significant logic errors identified: the light variable was only read once at startup and never updated; the else-if chain meant the freezing condition and temperature drop detection were unreachable; and the temperature drop check was inside the same chain as the rise check, preventing both from triggering. All were corrected by restructuring into separate if-blocks and moving sensor reads into the loop.

Week 5–6

Adaptive Model Implementation

Replaced the fixed 35°C threshold with an exponential moving average baseline. Dynamic thresholds calculated from the baseline. Fire risk score (0–4) introduced combining temperature and light. Warning margin adaptation added based on alertCount and allClearCount.

Week 7–8

Logging and Display Modes

Datalogger integrated with seven variables per reading. Adaptive log interval implemented (1–4 seconds by risk level). Button B display cycle (displayMode counter with modulo) added to show T, R, B, and TH readings on demand.

Week 9–10

Python Model and What-If Scenarios

Python model written to replicate the adaptive algorithm offline. Two what-if scenarios implemented: sudden heatwave (+8°C with freeze_baseline=True) and canopy loss (light ×1.4). Results compared in a final output table.

Week 11–12

Testing and Report

Final testing of all branches. Report website written. Video recorded demonstrating all features.

Testing

Testing was carried out throughout development. For the micro:bit, the temperature sensor was tested by warming the device with a hand to trigger the alert condition and checking that the alarm fired correctly. The Button A light modes were tested in a dark room (torch off), dim light (torch through paper), normal room light, and direct bright light. The rolling average was verified by watching the B: display value shift gradually over several minutes as ambient temperature changed, confirming the baseline was updating correctly rather than staying fixed.

For the Python model, the three scenarios were verified by checking that the baseline scenario produced a low average risk on realistic data, that the heatwave scenario with freeze_baseline produced a large number of critical events (confirming the frozen baseline was working correctly), and that the canopy loss scenario increased high-light events by the expected proportion.

A Problem Encountered

The most significant problem was the unreachable condition bug in the temperature display logic. The original code had a chain of else-if blocks checking: temperature > 35, then temperature < 35, then temperature < 0, then temperature < previoustemp − 3. Because temperature < 35 catches every value below 35, the checks for below zero and for temperature drops could never be reached — the second condition in the chain swallowed everything. The fix was to separate the conditions into independent if-blocks and reorder them so that the most specific check (below zero) came before the general check (below 35). This was identified by manually tracing through the code with test values (e.g. −2°C) and noting that the freezing icon never appeared.

The Python Model

The Python model (wildfire_model.py) processes a dataset of environmental readings — either exported from the micro:bit datalogger as a CSV, or a built-in simulated dataset if no file is present. It runs the same adaptive algorithm as the micro:bit: exponential moving average baseline, dynamic thresholds, fire risk scoring, and margin adaptation. The key function is run_model(), which accepts a temp_offset, a light_multiplier, and a freeze_baseline flag to configure each scenario.

Scenario 2 simulates deforestation removing tree canopy, increasing the direct sunlight reaching the forest floor and pushing more readings above the 170 light threshold, which adds to the fire risk score. The combined scenario shows that heatwave and canopy loss together produce a worst-case outcome significantly worse than either alone.

Python — core adaptive algorithm (run_model function)
# Exponential moving average — baseline shifts gradually
if not freeze_baseline:
    rolling_avg = rolling_avg * 0.9 + temp * 0.1

# Dynamic thresholds derived from live baseline
warning_threshold = rolling_avg + warning_margin
critical_threshold = rolling_avg + critical_margin

# Score fire risk relative to context, not hardcoded numbers
risk = 0
if temp > rolling_avg + 2:  risk += 1
if temp > warning_threshold: risk += 1
if temp > critical_threshold: risk += 1
if light > 170:             risk += 1

# Adapt margins based on sustained conditions (the feedback loop)
if risk >= 3:
    alert_count += 1
    if alert_count >= 4 and warning_margin > 2:
        warning_margin -= 1   # become more sensitive
elif risk == 0:
    all_clear_count += 1
    if all_clear_count >= 6 and warning_margin < 7:
        warning_margin += 1   # relax back toward normal
Section 05

Evaluation

The final artefact successfully meets all basic and advanced requirements of the brief. The embedded system collects real environmental data, simulates wildfire risk, and adapts its own sensitivity based on sustained conditions. The Python model extends this to a full dataset analysis with two meaningful what-if scenarios that produce clearly different and explainable outcomes.

The adaptive model performs well as a concept: the exponential moving average correctly shifts the baseline over time, and the margin adaptation demonstrably changes the system's behaviour under sustained alerts. The logged data includes the internal model state (warning_margin, rolling baseline, dynamic threshold), which makes the adaptation visible and verifiable — an examiner can open the CSV and see the margins change over the course of a run.

There are clear limitations. The micro:bit temperature sensor is affected by the heat generated by the device itself, which introduces a small upward bias in readings. The light sensor is affected by the angle at which the device is held. In a real deployment, calibrated external sensors would be used. Additionally, the current model uses temperature and light as proxies for wildfire risk, but a more complete model would also include humidity and wind speed — both of which significantly affect real fire spread.

Future Improvements

The most valuable next iteration would be to connect the micro:bit to external sensors via the GPIO pins — particularly a DHT22 humidity sensor and a soil moisture sensor. Humidity is arguably more important than temperature alone for wildfire risk assessment, since dry vegetation is what actually burns. Soil moisture would allow the model to simulate drought cycle detection, which is another key brief example.

The Python model could be extended with a time-series prediction component — using the rolling average trend to forecast where the baseline is heading and flag risk before it arrives, rather than only scoring current conditions. This would make it genuinely predictive rather than just descriptive, which would strengthen the advanced requirements further.
