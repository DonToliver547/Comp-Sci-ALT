1. Meeting the Brief

The following requirements from the project brief were addressed:

  • Embedded System Design The BBC micro:bit uses Button A and Button B as digital inputs, and the built-in temperature and light sensors as analogue inputs. The LED matrix and buzzer provide primary digital output. Button A displays a light level icon on demand; Button B shows the current temperature. The system starts automatically and runs continuously once powered on.
  • Data Collection Temperature and light level are read every cycle and written to the micro:bit datalogger every three seconds, producing a timestamped log of environmental readings throughout the session.
  • Process Simulation The system simulates wildfire risk in a forest environment. A fixed temperature threshold (35°C) triggers a critical alarm with an exclamation mark on the LED display and a repeating alarm tone. Temperatures below freezing display an X icon, and sudden changes of more than 3°C in a single cycle display a directional arrow to indicate a spike or drop.
  • Disaster Risk Modelling A Python model (wildfire_model.py) accepts sensor readings entered by the user, reconstructs a temperature and light data series from those values, and runs an adaptive wildfire risk algorithm across the full dataset, outputting results to CSV files.
  • What-If Simulations Two scenarios were tested: (1) a sudden +8°C heatwave with the baseline frozen to simulate an unexpected temperature event, and (2) a 40% increase in light intensity simulating canopy loss from deforestation.
  • Adaptive System The Python model uses an exponential moving average to maintain a live baseline of normal temperature. Warning and critical thresholds are derived from this baseline rather than hardcoded. Warning margins tighten after sustained high-risk readings and relax after a period of calm, giving the model a feedback loop that adjusts its own sensitivity over time.

2. Investigation

Research began with understanding how real-world wildfire early warning systems work. The Bosch Forest Fire Detection System [1] uses a network of environmental sensors monitoring temperature, humidity, and gas concentration. Unlike simple threshold-based alarms, these systems use models trained on historical data to detect anomalous conditions before a fire spreads. This was a key insight: reacting to a fixed temperature is not the same as detecting danger relative to context.

The WWF Guianas early warning network [2] emphasises how detecting rate of change matters more than absolute values. A forest that is normally warm has a different danger profile to one that is usually cool — the same temperature could be safe in one and critical in the other. This directly influenced the decision to include sudden change detection in the embedded system, where a rise or fall of more than 3°C in a single cycle triggers a directional arrow even if the overall risk level has not changed.

Research into existing micro:bit environmental projects [3] confirmed that the built-in temperature and light sensors, while not research-grade, are sufficient for simulating environmental monitoring logic. Several examples used fixed thresholds (e.g. "if temperature > 30, alert") which is the approach used in the embedded system here, with the more sophisticated adaptive logic reserved for the Python model where it can operate over a full dataset.

The EU Climate Change website [4] provided context on how deforestation and rising temperatures interact with wildfire risk — specifically that canopy loss increases direct sunlight reaching the forest floor, drying ground vegetation and raising fire probability. This directly motivated the second what-if scenario in the Python model, which simulates deforestation by multiplying light levels by 1.4.

Based on this research, the key design decisions were: to use a fixed threshold on the embedded device for simplicity and reliability, to add sudden-change detection as a proxy for the rate-of-change insight from WWF research, and to implement the full adaptive model in Python where it could be applied to a complete dataset rather than one reading at a time.

3. Plan and Design

Design Objectives

The system needed to meet all basic and advanced requirements while remaining achievable on a BBC micro:bit — a device with no network connectivity, limited memory, and a 5×5 LED display. The core design objective was to build a functional embedded monitor that could collect real environmental data and feed it into an offline Python model capable of adaptive analysis and scenario testing.

Options Considered

Initially, the embedded system was considered as the location for the full adaptive model. This was rejected because the micro:bit has a very limited processing loop and maintaining rolling averages and margin counters across two concurrent forever loops introduced timing complexity. Instead, the embedded system uses a straightforward fixed threshold (35°C for critical, below 0°C for frost) with sudden-change detection, while all adaptive logic lives in the Python model. This separation keeps each component reliable and testable independently.

A Raspberry Pi was considered as an alternative platform but rejected — the micro:bit is sufficient for the sensors required and is more relevant to the embedded systems brief.

Stakeholders and End User Needs

The primary end user is a forest ranger or environmental monitoring officer who needs to be alerted quickly and clearly. This shaped two decisions: the LED display uses simple iconic symbols readable at a glance (exclamation mark for critical heat, X for frost, arrows for sudden changes), and the alarm is audible. A secondary user is a data analyst who would use the logged CSV data and Python model outputs to review historical risk patterns. Their need is for well-labelled, complete data — which is why the Python model logs seven variables per reading including internal model state.

Technologies

The embedded system uses MakeCode TypeScript on the BBC micro:bit, providing built-in access to the temperature sensor, light sensor, LED matrix, buzzer, buttons, and datalogger. The computer model is written in Python 3 using only the standard library (csv, math) so it can be run without installing any packages.

System Architecture

The embedded system has three concurrent sections on the micro:bit: a main forever loop that reads sensors, updates the display, and detects sudden changes; an adaptive logging loop that writes temperature and light to the datalogger every three seconds; and two interrupt-driven button handlers. These feed into an offline Python model:

Embedded System Logic

The main loop reads temperature and light each cycle, then evaluates three independent conditions using separate if-blocks: whether temperature exceeds 35°C (critical alarm), whether it is below 0°C (frost icon), and whether it has risen or fallen more than 3°C since the previous reading (spike or drop arrow). Using separate blocks rather than an else-if chain ensures that no condition can be masked by another — a critical insight discovered during a debugging phase described in Section 4.

4. Create

Development Milestones

Week 1–2

Basic Sensor Reading and Display

Initial micro:bit code read temperature and displayed it on Button B press. Light level reading added. A simple threshold alert (temperature above 35°C triggers an exclamation mark and alarm tone) was implemented and tested on the device.

Week 3–4

Logic Bug Fixes

Three significant logic errors were identified and corrected. The light variable was only being read once at startup rather than inside the loop. The original else-if chain made the below-zero (frost) condition and the temperature-drop detection unreachable. Restructuring into separate if-blocks and moving all sensor reads into the main loop resolved these issues.

Week 5–6

Button Handlers and Display Icons

Button A was implemented to read the current light level and display a contextual icon: a moon shape for darkness, a half sun for dim light, a sun outline for normal daylight, and a full blazing sun for high sunlight (above 170). Button B was configured to display the current temperature as a number. All LED patterns were tested in different lighting conditions.

Week 7–8

Logging Loop and Sudden Change Detection

A second forever loop was added to write light value and temperature to the micro:bit datalogger every three seconds independently of the main loop. Sudden-change detection was added to the main loop: a rise or fall of more than 3°C compared to the previous reading shows a directional arrow for one second.

Week 9–10

Python Model and What-If Scenarios

The Python model was written to accept sensor summary values entered by the user, reconstruct a full data series mathematically, then run the adaptive algorithm across it. Two what-if scenarios were implemented: a sudden heatwave (+8°C with freeze_baseline=True) and canopy loss (light ×1.4). Results are saved to four CSV files and printed as a comparison table.

Week 11–12

Testing and Report

Final testing of all branches on the device. Report written and structured as an HTML file. Video recorded demonstrating all features of both the embedded system and the Python model.

Testing

For the embedded system, the temperature sensor was tested by warming the device with a hand to verify the 35°C alarm fired correctly with the exclamation icon and tone. The below-zero condition was tested by noting that holding an ice pack near the device caused the X icon to appear. Button A light modes were tested in a dark room, dim light, normal room light, and in direct sunlight near a window. Sudden-change detection was verified by briefly warming the sensor quickly to confirm the upward arrow appeared for one second before reverting to the normal display.

For the Python model, the three scenarios were verified by checking that a realistic baseline dataset produced low average risk, that the heatwave scenario with freeze_baseline generated a large number of critical events, and that the canopy loss scenario increased the proportion of high-light readings by approximately the expected amount.

A Problem Encountered

The most significant problem was an unreachable condition bug in the temperature display logic. The original code used an else-if chain: temperature above 35, else temperature below 35, else temperature below 0, else temperature drop. Because the "temperature below 35" branch catches every value below 35, the checks for below zero and for temperature drops could never be reached — the code would always stop at the second branch. The frost icon and drop arrow never appeared during testing, which was how the bug was noticed. The fix was to restructure all conditions into independent if-blocks and place the most specific check (below zero) before any general check. After restructuring, all four display states were confirmed to work correctly by tracing through with test values of −2°C and a rapid 4°C drop.

The Python Model

The Python model is divided into four sections. The first collects summary values from the user — peak temperature, lowest temperature, starting temperature, largest single-cycle change, peak light, average light, number of readings, and session duration. The second reconstructs a full data series from these values using a sine-wave interpolation, so the model has a realistic curve of readings to process even without a CSV export from the device. The third section runs the adaptive wildfire risk algorithm: it maintains an exponential moving average baseline (updated each cycle as rolling_avg = rolling_avg * 0.9 + temp * 0.1 unless freeze_baseline is set), derives warning and critical thresholds from the baseline, scores fire risk from 0 to 4 by checking temperature against the baseline and thresholds and light against 170, and adapts the warning margin based on sustained high-risk or all-clear counts. The fourth section saves the per-reading results to CSV and prints a comparison table across all four scenarios.

Python — core adaptive algorithm (from run_model)
# Exponential moving average — baseline shifts gradually
if not freeze_baseline:
    rolling_avg = rolling_avg * 0.9 + temp * 0.1

# Dynamic thresholds derived from live baseline
warning_threshold  = rolling_avg + warning_margin
critical_threshold = rolling_avg + critical_margin

# Score fire risk relative to context, not hardcoded numbers
risk = 0
if temp > rolling_avg + 2:    risk += 1
if temp > warning_threshold:  risk += 1
if temp > critical_threshold: risk += 1
if light > 170:               risk += 1

# Adapt margins based on sustained conditions (the feedback loop)
if risk >= 3:
    alert_count += 1
    all_clear_count = 0
    if alert_count >= 4 and warning_margin > 2:
        warning_margin -= 1   # become more sensitive
elif risk == 0:
    all_clear_count += 1
    alert_count = 0
    if all_clear_count >= 6 and warning_margin < 7:
        warning_margin += 1   # relax back toward normal

5. Evaluation

The final artefact successfully meets all basic and advanced requirements of the brief. The embedded system collects real environmental data, simulates wildfire risk using a threshold-based alarm with contextual light and sudden-change detection, and logs readings to the datalogger. The Python model extends this to a full dataset analysis using an adaptive algorithm, with two meaningful what-if scenarios that produce clearly different and explainable outcomes.

The separation of concerns between the embedded system and the Python model worked well. The micro:bit code is simple and reliable — all conditions are independent if-blocks, sensor reads happen every cycle, and the logging loop runs separately so that display and data collection do not interfere with each other. The Python model handles the complexity of the adaptive algorithm where it can operate on a full sequence of readings rather than one at a time.

There are clear limitations. The micro:bit temperature sensor is affected by heat generated by the device itself, introducing a small upward bias in readings. The light sensor is affected by the angle at which the device is held. The embedded alarm threshold is fixed at 35°C and does not adapt to ambient conditions — a genuinely cool forest and a warm one would trigger at the same absolute temperature. In a real deployment, the adaptive logic from the Python model would ideally run on the device itself, or calibrated external sensors would be used. The current model also uses temperature and light as proxies for wildfire risk; a more complete model would include humidity and wind speed, both of which significantly affect real fire spread.

Future Improvements

The most valuable next step would be to connect external sensors via the GPIO pins — particularly a DHT22 humidity sensor. Humidity is arguably more important than temperature alone for wildfire risk, since dry vegetation is what actually burns. Adding humidity would allow the Python model to incorporate a fire weather index, a standard real-world metric combining temperature, humidity, and wind. The Python model could also be extended with a time-series prediction component — using the rolling average trend to forecast where the baseline is heading and flag risk before it arrives, rather than only scoring current conditions.
